Fiscal Year Variants
›

El FYV define la estructura del año fiscal: períodos normales + períodos especiales (cierre).

Calendar Year (ej. K4)

Enero–Diciembre. El año fiscal = año calendario. Same structure every year.

Non-Calendar Year

Fiscal year cruza dos años civiles (ej. Abril–Marzo). Requiere Year Shift en la tabla de períodos.

Posting PeriodsNormalmente 12 (uno por mes) Special Periods1–4 — para ajustes de cierre (audit, tax, etc.) Year ShiftPara FYV no-calendario: -1 en períodos del año anterior, 0 en el año actual
⚠️ Restricción clave: Todos los Company Codes de un Controlling Area deben tener el mismo Fiscal Year Variant. No se puede cambiar de forma individual.

G/L Account Master Data — Dos Segmentos
›

Cada cuenta GL tiene dos segmentos que se deben crear:

Chart of Accounts Segment

Válido para todos los CoCd que usen ese CoA. Contiene: G/L Account Type, Account Group, Short/Long Text, Group Account Number.

Company Code Segment

Específico por CoCd. Contiene: Account Currency, Tax Category, Sort Key, Field Status Group, datos de CO (Cost Element Category).

Field Status — Prioridad y Control
›

El Field Status determina si un campo es obligatorio, opcional, oculto o display-only. Se aplica en master data y en documentos.

Prioridad (de mayor a menor):
1. Suppress/Hide  ›  2. Display  ›  3. Required  ›  4. Optional

Cuando dos reglas entran en conflicto, gana la de mayor prioridad. Por eso si el Sort Key es "Display" en la Activity "Change Customer" y "Optional" en el Account Group, el campo quedará no editable en cambio.

Sensitive Fields (Dual Control)

Un campo definido como Sensitive requiere confirmación por otro usuario tras ser modificado en el BP/Vendor. Hasta que se confirma, el cambio queda pendiente.

Document Splitting — Clasificación de Document Types
›

Con el Document Splitting activo, cada nuevo Document Type debe ser clasificado para que el sistema sepa cómo distribuir las posiciones del documento entre los objetos de segmento (Profit Center, Segment).

IMG PathFA → General Ledger → Business Transactions → Document Splitting → Classify Document Types for Document Splitting Business Transaction0300 = Vendor Invoice · 0200 = Customer Invoice (Outgoing) Variant0001 = Standard
Si un Document Type no está clasificado y el Document Splitting está activo, no se podrá postear con ese document type.

Field Status Groups y Field Status Variants
›

El Field Status Group (FSG) controla el comportamiento de los campos en las líneas de posting (no en el master data). Se asigna en el Company Code Segment de la cuenta GL.

Field Status VariantContenedor de Field Status Groups. Se asigna al Company Code. Ejemplo: 0010 Field Status GroupAgrupa campos por comportamiento (Required/Optional/Hide) para un tipo de cuenta. Ej: YB01, YB03, FS## IMG PathFA Global Settings → Ledgers → Fields → Define Field Status Variants Asignación CoCdFA Global Settings → Ledgers → Fields → Assign Company Code to Field Status Variants
Combinación de Field Status: El sistema combina el FSG de la cuenta + el Posting Key. La prioridad aplica: Suppress > Display > Required > Optional.

Posting Control

Ejercicios 18–25 — Document Splitting, Reversals, Tax Codes, Cross-Company Code

📌 Key Exam Points

Document Splitting es un tema conceptualmente profundo. Estudia la lógica de derivación (Business Transaction + Base Line). Tax codes: diferencia Input vs Output. Cross-Company: clearings automáticos.

✂️
Document Splitting — Concepto y Lógica
›

El Document Splitting permite distribuir automáticamente posiciones contables (receivables, tax) entre múltiples objetos de segmento según las líneas de ingresos/gastos.

Cómo funciona
  • Base Lines (ej. Revenue 41000500): posiciones con asignación explícita de Profit Center/Segment
  • Split Lines (ej. Receivable 12100000, Tax 22000000): se distribuyen proporcionalmente según las base lines
  • El sistema usa la clasificación del Document Type (Business Transaction + Variant) para identificar qué es base y qué se split
Entry ViewLo que el usuario introduce: sin splitting visible GL ViewVisión del libro mayor: incluye las posiciones spliteadas por segmento/PC Expert ModeEn la simulación GL View, muestra: Business Transaction, Variant, Acct Category de base lines y split lines
Para el examen: BT 0200 = Customer Invoice. BT 0300 = Vendor Invoice. Variant 0001 = Standard.
↩️
Document Reversal y Negative Postings
›

Para revertir un documento incorrecto, SAP ofrece dos métodos:

Normal Reversal

Crea un nuevo documento con importes opuestos. Los saldos de cuenta se incrementan en la dirección contraria.

Negative Posting

Permite que el documento de reversión aparezca como resta en el mismo lado del mayor. Requiere activar "Negative Postings Permitted" en el Company Code.

Activar neg. postingsFA → General Ledger → Business Transactions → Adjustment Posting/Reversal → Permit Negative Posting Reason CodesAl revertir se selecciona un motivo. Los reason codes con indicador "Neg.Pstg" realizan la reversión negativa (ej. 03, 04, 05). Fiori AppDesde Manage Journal Entries → seleccionar documento → Reverse
💲
Terms of Payment (Condiciones de Pago)
›

Las Terms of Payment definen los plazos y descuentos por pronto pago (cash discounts). Se asignan al master record del BP (Customer/Vendor).

Baseline DatePunto de partida para calcular la fecha de vencimiento. Puede ser: Posting Date, Document Date, Entry Date, o una fecha fija. TérminosHasta 3 tramos: % de descuento + días. Último tramo = 0% (pago neto). Account TypeSe puede restringir a Customers, Vendors, o ambos. IMG PathFA → AR/AP → Business Transactions → Incoming Invoices → Maintain Terms of Payment
Ejemplo: 5% inmediato, 2%/14 días, neto/30 días con Baseline Date = Posting Date.

Payment Differences — Partial Payment vs Residual Item
›
Partial Payment

Se registra el pago parcial. Tanto la factura original como el pago parcial quedan como partidas abiertas. El pago se "referencia" a la factura pero no la cierra.

Residual Item

La factura original se cierra completamente. Se crea una nueva partida abierta por la diferencia. Se puede asignar un Reason Code para clasificar el motivo.

Reason Codes
IMG PathFA → AR/AP → Business Transactions → Incoming Payments → Incoming Payments Global Settings → Overpayment/Underpayment → Define Reason Codes Indicador C"Charge Off Differences via Separate Account" — el importe de la diferencia se lleva a una cuenta específica (ej. cuenta de pérdidas/mermas). Si no se marca, va a la cuenta de clearing estándar.
Ejemplo Ejercicio 28: Factura 300.000 EUR, pago 250.000 EUR. Diferencia 50.000 EUR → Residual Item con Reason Code Z## (daños en tránsito). La diferencia se lleva a cuenta 44000000.

El Variant Principle — Patrón que debes dominar
›

SAP aplica el Variant Principle en múltiples objetos de configuración para reutilizarlos en varios Company Codes:

  • Fiscal Year Variant → se crea una vez, se asigna a varios Company Codes
  • Posting Period Variant → una variante de períodos, múltiples Company Codes
  • Field Status Variant → un conjunto de FSG, múltiples Company Codes
  • Tolerance Group → una variante por tipo de usuario, se asigna a usuarios individuales
  • Chart of Accounts → un CoA usado por múltiples Company Codes
Lógica: Define una vez, usa muchas veces. Cambiarlo en la variante afecta a todos los Company Codes asignados.
